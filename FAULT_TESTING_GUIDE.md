# NetBox AIOps Agent - 故障注入测试指南

## 🎯 测试目标

验证系统的完整能力链：
```
异常发现 → 诊断分析 → Tool Calling → 自动处置 → 生成报告
```

## 🚀 快速开始

### 第一步: 启动故障注入工具

```bash
python fault_injection_test.py
```

你会看到这个菜单：
```
  1. 🛑 停止 NetBox 容器 (验证服务异常检测)
  2. 📁 模拟磁盘空间不足 (创建大文件)
  3. ⚙️  修改 docker-compose.yml (配置错误检测)
  4. 🔄 制造容器频繁重启 (循环重启容器)
  5. 🔐 删除/修改环境变量 (配置缺失检测)
  6. 🧪 一键注入所有故障 (极限测试)

  9. ✅ 恢复所有故障 (清理测试环境)
  0. ❌ 退出
```

## 🧪 测试场景详解

### 场景 1: 服务异常检测（推荐先试这个）

**操作**: 选择菜单 `1` - 停止 NetBox 容器

**注入的故障**:
```
docker stop netbox-docker-netbox-1
```

**预期系统行为**:
1. ✅ `check_service_health` 检测到容器状态为 `exited`
2. ✅ 诊断引擎标记为 **HIGH** 优先级问题
3. ✅ 自动建议 `inspect_container_logs` 查看崩溃原因
4. ✅（如启用自动修复）执行 `restart_service` 重启服务

**验证命令**:
```bash
# 运行完整诊断
python main.py --mode full

# 或用 AI Agent 模式
python main.py --mode ai --provider doubao
```

**预期输出**:
```
检测到 1 个问题:
  - 容器 netbox-docker-netbox-1 状态异常: exited

诊断结果:
  严重级别: HIGH
  根因: 应用服务已停止
  建议: 检查容器日志，重启服务
```

---

### 场景 2: 磁盘空间不足检测

**操作**: 选择菜单 `2` - 创建 512MB 测试文件

**注入的故障**:
- 在当前目录创建一个大文件 `large_test_file.tmp`
- 模拟磁盘使用率上升

**预期系统行为**:
1. ✅ `check_disk_space` 检测到磁盘使用率
2. ✅ 如果超过阈值（默认 85%）触发告警
3. ✅ 建议 `prune_docker_resources` 清理 Docker 资源
4. ✅（如启用自动修复）执行资源清理

**验证命令**:
```bash
python main.py --mode full
```

---

### 场景 3: 配置文件错误检测

**操作**: 选择菜单 `3` - 修改 docker-compose.yml

**注入的故障**:
- 注释掉 `restart: always` 策略
- （或）注入无效配置项

**预期系统行为**:
1. ✅ `analyze_docker_compose` 解析配置文件
2. ✅ 在 warnings 中提示"服务缺少 restart 策略"
3. ✅ 建议检查并修复配置文件

---

### 场景 4: 容器频繁重启检测

**操作**: 选择菜单 `4` - 循环重启 redis 容器 3 次

**注入的故障**:
- 容器重启计数 > 0

**预期系统行为**:
1. ✅ `check_service_health` 检测到 `restart_count > 0`
2. ✅ 诊断引擎标记为"容器频繁重启"问题
3. ✅ 建议 `inspect_container_logs` 检查崩溃原因

---

### 场景 5: 环境配置缺失检测

**操作**: 选择菜单 `5` - 注释掉 POSTGRES_PASSWORD

**注入的故障**:
- .env 文件中缺失必要的环境变量

**预期系统行为**:
1. ✅ `validate_env_config` 检测到缺失配置
2. ✅ 标记为 **CRITICAL** 级别问题
3. ✅ 建议恢复配置文件

---

### 场景 6: 极限测试（所有故障一起上）

**操作**: 选择菜单 `6` - 一键注入所有故障

**注入的故障**:
```
1. 停止 NetBox 容器
2. 创建大文件模拟磁盘不足
3. 修改 docker-compose.yml 配置
4. 制造容器频繁重启
5. 注释掉 POSTGRES_PASSWORD
```

**这是对系统的终极考验！**

预期能检测到 5+ 个问题，并按严重程度分级排序。

## 💡 高级测试技巧

### 技巧 1: 对比模式
```bash
# 1. 先测健康状态
python main.py --mode full > test_healthy.log

# 2. 注入故障
# 3. 再测故障状态
python main.py --mode full > test_fault.log

# 4. 对比差异
diff test_healthy.log test_fault.log
```

### 技巧 2: 观察 Tool Calling 流程
在 AI Agent 模式下，仔细观察：
```
迭代 1: 调用什么工具？为什么？
迭代 2: 下一步决策是什么？
迭代 3: 是否调用了修复动作？
...
迭代 N: 何时生成最终报告？
```

**好的 Tool Calling 应该是**:
- ✅ 逻辑连贯，逐步深入
- ✅ 先检测，再分析，最后处置
- ✅ 不会重复调用相同工具
- ✅ 问题解决后及时停止

### 技巧 3: 关闭自动修复，观察纯诊断
```bash
python main.py --mode full --no-auto-remediate
```
观察系统给出的纯诊断建议是否合理。

## ❓ 常见问题

**Q: 注入故障后容器真的停了吗？**
```bash
docker ps  # 查看运行中的容器
docker ps -a  # 查看所有容器
```

**Q: 怎么看系统检测到了哪些问题？**
- 在 `--mode full` 模式下会打印所有检测到的问题
- 在 `--mode ai` 模式下可以看到完整的 Tool Calling 过程

**Q: 自动修复会破坏我的真实环境吗？**
- 不会！安全动作（检测类）总是执行
- 处置动作（重启、清理）有条件保护
- 你也可以用 `--no-auto-remediate` 关闭自动修复

**Q: 怎么知道系统的处理是否正确？**
- 对比你注入的故障和检测到的问题
- 检查诊断建议是否符合预期
- 验证处置动作是否解决了问题
